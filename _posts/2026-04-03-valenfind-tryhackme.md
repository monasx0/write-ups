# Valenfind - TryHackMe Write-up
![1.png](/assets/img/writeups/valenfind/1.png)
**Platform:** [TryHackMe](https://tryhackme.com)<br>**Difficulty:** Easy<br>**Category:** Web Application Pentesting

---
## Summary
[Valenfind](https://tryhackme.com/room/lafb2026e10) is a Valentine-themed love matching web app that was "vibe coded" — meaning it was built quickly with AI assistance and little security review. This room walks through a series of web vulnerabilities including information disclosure and Local File Inclusion (LFI), ultimately leading to admin database access and flag retrieval.

---
## Step 1 — Reconnaissance & Registration

Visiting the target IP, I was greeted with a Valentine-themed website with a **Login** and **Register** page.
![1.png](/assets/img/writeups/valenfind/2.png)

I first tested the login page for SQL injection — it wasn't vulnerable. So I moved on and registered a new account with dummy information:

- Real Name
- Email Address
- Phone Number
- Home Address
- Bio / About Me

I filled in random/fake details and successfully logged in.
![1.png](/assets/img/writeups/valenfind/3.png)

---

## Step 2 — Exploring the Website

Once logged in, I browsed other users' posts. One post stood out immediately:<br>**User:** `cupid`<br>**Post description:** *"I keep the database secure. No peeking."*
![1.png](/assets/img/writeups/valenfind/4.png)

This felt like a hint left by the room creator pointing toward something sensitive. I right-clicked and viewed the page source.

---

## Step 3 — Discovering the Vulnerable Endpoint

In the page source, I found a JavaScript snippet dynamically fetching a layout file using this URL pattern:
````
/api/fetch_layout?layout=${layoutName}
````

This is a classic sign of a **Local File Inclusion (LFI)** vulnerability — the server is reading files based on user-supplied input without proper validation.

---

## Step 4 — Exploiting LFI

I tested the endpoint with a basic path traversal payload to read the Linux password file:
````
/api/fetch_layout?layout=../../../../etc/passwd
````

![1.png](/assets/img/writeups/valenfind/5.png)
It worked. The server returned the contents of `/etc/passwd`, confirming the LFI vulnerability.

---

## Step 5 — Identifying the Web Server Application

To learn more about what's running the server, I read a Linux pseudo-file that reveals the command used to start the current running process:
````
/api/fetch_layout?layout=../../../../proc/self/cmdline
````
**Output:**
````
/usr/bin/python3/opt/Valenfind/app.py
````
This tells us:
- The web server is running with **Python 3**
- The main application file is located at `/opt/Valenfind/app.py`

> `/proc/self/cmdline` is a special Linux file that stores the exact command used to launch the currently running process. It's not a real file on disk — it's generated by the kernel in real time.

---

## Step 6 — Reading the Application Source Code

Since we know the app file path, I used LFI to read it directly:
````
/api/fetch_layout?layout=../../../../opt/Valenfind/app.py
````



The source code was exposed. Inside, I found something very interesting — a **hardcoded admin API key**: `CUPID_MASTER_KEY_2024_XOXO`<br>This key is meant to be used with the `X-Valentine-Token` header to access an admin-only endpoint: `/api/admin/export_db`

![1.png](/assets/img/writeups/valenfind/6.png)

---

## Step 7 — Accessing the Admin Database

Using `curl`, I crafted a request with the stolen API key to hit the admin export endpoint:
```bash
$ curl -H "X-Valentine-Token: CUPID_MASTER_KEY_2024_XOXO" http://10.49.186.65:5000/api/admin/export_db --output output.txt
```

This saved the database export to `output.txt`.

---

## Step 8 — Extracting the Flag

When I tried to read the file normally, some of the data was in binary format. A plain `grep` for the flag gave this warning:
```bash
$ cat output.txt | grep "THM"
grep: (standard input): binary file matches
```
This means the flag exists but grep skipped it because of binary data. The fix is to use the `-a` flag which tells grep to treat binary files as text:
```bash
cat output.txt | grep -a "THM"
```

![1.png](/assets/img/writeups/valenfind/7.png)

- [x] Flag retrieved successfully.
